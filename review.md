# MathBridge AI — Code Review Findings

_Last updated: 2026-06-22_

Review of the project at `Math_tutor_app_0620`, answering the nine questions

## 1) What OpenAI models are used?

A single OpenAI chat model is used everywhere. It is configured centrally and
re-instantiated on every call (no embeddings, no image, no audio models).

| Model | Where it is set | Where it is consumed |
|-------|-----------------|----------------------|
| `gpt-4.1-mini` | `config.py:11` (`LLM_MODEL = "gpt-4.1-mini"`, `LLM_TEMPERATURE = 0.3`) | `ai/tutor_chain.py:175-179` (tutor reply) and `ai/signals.py:30-34` (signal extractor, with `temperature=0`) |

Both call sites use `langchain_openai.ChatOpenAI` and read the API key from the
`OPENAI_API_KEY` environment variable.

No other model identifiers appear in the codebase (verified — only `LLM_MODEL`
is imported from `config`).

## 2) How are visual explanations generated? What model?

**No model is used.** Visuals are deterministic PIL drawings — they are NOT
generated by an LLM and NOT generated by an image model such as DALL·E.

Flow:

1. The tutor LLM (`gpt-4.1-mini`) is prompted in `ai/tutor_chain.py:13-54` to
   return a JSON payload with `needs_visual`, `visual_type`, and `visual_data`.
   The allowed visual types listed in the prompt are very narrow:
   `"none"` and `"fraction_division_bar"` (`ai/tutor_chain.py:57-60`).
2. `ui/student.py:283-405` then runs a **regex fallback** over the student's raw
   text to detect five additional patterns (fraction division, ratio, miles/hr,
   cost-per-item, fraction-decimal equivalence) and overrides `visual_type` /
   `visual_data` accordingly. So in practice, the regex layer — not the LLM —
   chooses the visual in most cases.
3. The chosen `(visual_type, visual_data)` is passed to
   `ai/visual_renderer.create_visual()` (`ai/visual_renderer.py:54-101`), which
   dispatches to one of four hand-coded PIL renderers:
   - `_create_fraction_division_bar` (`ai/visual_renderer.py:104`)
   - `_create_ratio_bar` (`ai/visual_renderer.py:253`)
   - `_create_unit_rate_table` (`ai/visual_renderer.py:323`)
   - `_create_fraction_decimal_bar` (`ai/visual_renderer.py:380`)

Each renderer draws rectangles/text with `PIL.ImageDraw` and returns a
`PIL.Image`. The image is displayed by `ui/student.py:93-107` via
`st.image(...)` but never written to disk.

**Inconsistency worth noting:** the prompt only authorizes
`"fraction_division_bar"` (`ai/tutor_chain.py:57-60`), but the renderer
supports four types and the regex layer can set ratio / unit-rate / fraction-
decimal types. The LLM does not know these visual types exist; only the regex
shortcut path can trigger them.

## 3) How is the student signal modeled and analyzed? What data / model?

**Model:** the same `gpt-4.1-mini` chat model from `config.LLM_MODEL`,
instantiated in `ai/signals.py:30-34` with `temperature=0` to keep extractions
stable.

**Data fed to the model:** the last six messages of the current chat session
(`ai/signals.py:25-29`). Nothing else — no curriculum context, no mastery
history, no prior sessions. If history is shorter than 2 messages, an empty
dict is returned (`ai/signals.py:22-23`).

**Schema requested** (`ai/signals.py:7-18`):

```json
{
  "concept": "...",
  "misconception": "... or null",
  "hints_used": "X hint(s) requested",
  "engagement": "Active | Moderate | Low | Needs scaffold",
  "next_support": "... or null"
}
```

**How signals are consumed (this is where the heuristic — not the LLM —
takes over):**

- Per turn, `ui/student.py:435-441` writes the parsed dict to
  `st.session_state.signals` and persists it via
  `memory.student_memory.update_signals` (`memory/student_memory.py:44-59`),
  which also appends to `signals_history` with a timestamp.
- On session end, `memory/student_memory.update_mastery`
  (`memory/student_memory.py:122-182`) converts the signals into a mastery
  score delta:
  - `_match_mastery_skill` (`memory/student_memory.py:185-247`) maps the
    LLM's free-text `concept` onto one of four fixed skill buckets
    (`Fraction comparison`, `Equivalent fractions`, `Fraction addition`,
    `Word problems`) using a hardcoded keyword table, with a substring fallback.
  - `_calculate_mastery_delta` (`memory/student_memory.py:250-312`) computes
    a delta in `[-3, +3]`:
    - `misconception` present → −2, absent → +2
    - 0 hints → +1, ≥3 hints → −1
    - engagement word "active/engaged/persistent" → +1; "frustrated/low/
      disengaged/needs scaffold" → −1
  - Score clamped to `[0, 100]` (`memory/student_memory.py:156`).

So the "model" of the student is: **LLM-extracted signals → keyword bucket →
small hand-tuned integer delta on a 0–100 score**, persisted in
`data/memory/<student_id>.json` along with `signals_history` and
`mastery_history`.

## 4) How are teacher support insights generated? What analysis, on what data?

**There is no separate LLM call for "teacher insights".** The teacher view is a
read-only rendering of data already produced by the tutor pipeline plus a few
seeded strings.

File: `ui/teacher.py`. Inputs come from
`memory.student_memory.load_student(student_id)`.

| Card / panel | Source | How it is computed |
|--------------|--------|--------------------|
| Topic Focus (`ui/teacher.py:115-118`) | `signals["concept"]` or `current_topic` | Direct read from the latest LLM signal dict |
| Mastery Trend (`ui/teacher.py:120-126`) | `mastery_history[-1]` | Latest delta from `update_mastery` |
| Support Needed (`ui/teacher.py:127-131`) | `mastery` dict | `min(mastery, key=mastery.get)` — the lowest-scoring skill bucket |
| Engagement (`ui/teacher.py:132-136`) | `signals["engagement"]` | Direct read of the LLM label |
| Possible Misconception, Hint Usage, Next Support (`ui/teacher.py:142-176`) | `signals["misconception"]`, `signals["hints_used"]`, `signals["next_support"]` | Direct reads |
| Student Mastery Map (`ui/teacher.py:186-208`) | `student["mastery"]` | Per-skill bar, traffic-light bucketed: ≥75 green, ≥50 yellow, else red |
| Key Insights & Suggested Actions (`ui/teacher.py:210-243`) | `signals["misconception"]`, `signals["next_support"]`, plus `student["insights"]` (seeded list) | Concatenates dynamic items from the latest signals with **seeded** insights from `_default()` in `memory/student_memory.py:398-420` |
| Recent Signal History (`ui/teacher.py:251-280`) | `signals_history[-5:]` | Direct display |
| Recent Mastery Updates (`ui/teacher.py:284-381`) | `mastery_history[-5:]` | Direct display — note this block is **duplicated**: lines 284-331 and 333-381 render the same data twice |

**Summary:** the "teacher analysis" is (a) the LLM-extracted signals from
question 3 plus (b) the heuristic mastery deltas from question 3, formatted for
display. There is no second analytical pass over the data. The "insights" panel
mixes per-turn LLM output with three hardcoded default insights authored in
`_default()` (`memory/student_memory.py:398-420`).

## 5) Where is the configuration that makes the tutor "give answers directly"?

The system prompt in `ai/tutor_chain.py:13-54` (the `_SYSTEM` string) controls
tutor behavior. The relevant guidelines as written today:

```text
Tutoring guidelines:
- Guide students with questions, hints, and short explanations.
- Do not overwhelm the student with a long answer.
- Use language appropriate for a Grade 6 student.
- Celebrate effort and gently redirect mistakes.
- Use concrete examples such as fraction bars, tape diagrams, number lines,
  measuring cups, and equal groups.
- When curriculum notes are provided, align the explanation with them.
- If curriculum notes are not relevant, do not force them into the answer.
- Keep the answer concise, usually 2 to 4 sentences.
```

The prompt **says** "guide students with questions, hints, and short
explanations", but several things push the model toward giving the answer
outright:

1. **No "do not reveal the answer" / "ask one question first" rule.** The
   prompt never tells the model to withhold the solution or to use a
   Socratic step-by-step pattern. "Guide with questions" is a hint, not a
   constraint.
2. **The required JSON schema (`ai/tutor_chain.py:42-51`) has an `answer`
   field**, not a `next_question` / `hint` / `step` field. The model is
   literally being asked for "A short student-facing explanation in English"
   on every turn.
3. **The quick-action buttons** in `ui/student.py:111-138` send literal
   prompts like `"Can you show me the steps to solve this?"` and
   `"Give me a similar practice problem."` — both directly request the
   answer/steps.
4. **Temperature is 0.3** (`config.py:12`, applied at
   `ai/tutor_chain.py:177`), so once the prompt nudges toward "explain", the
   model is consistent about explaining.
5. The post-processing in `_parse_json_response` (`ai/tutor_chain.py:98-162`)
   pulls only the `answer` field for display — there is no path for a
   model-emitted "hint" or "follow-up question" to render differently.

**To change this behavior**, edit `_SYSTEM` in `ai/tutor_chain.py:13-54`
to add explicit Socratic rules (e.g. "never reveal the final numeric
answer on the first turn", "always end with a single question for the
student"), and consider extending the JSON schema to include an explicit
`next_question` field plus a `reveal_answer` boolean gated by hint count
from the signals.

## 6) Is curriculum grounding keyword-only? What expansion paths exist?

**Yes — it is purely lexical, no embeddings, no vector store.**

Implementation is in `ai/rag.py` (the filename is "rag.py" but the technique is
a hand-tuned BM25-like keyword scorer):

- `_normalize_text` (`ai/rag.py:67-74`): lowercases, expands `÷`/`×`.
- `_stem_token` (`ai/rag.py:77-102`): tiny hand-written stemmer (drops `-ies`,
  `-ing`, `-ed`, `-es`, `-s`) plus an `ALIASES` dict (`ai/rag.py:30-64`).
- `_tokenize` (`ai/rag.py:105-126`): regex + stop-word filtering, with
  stop-list at `ai/rag.py:14-27`.
- `_score_document` (`ai/rag.py:129-196`): per-document score using
  filename-token boost (×12), capped body frequency (≤6), a hardcoded
  `concept_boosts` table for nine domain phrases such as
  `{"reciprocal","algorithm"}` → +20, and a `lesson<N>` filename match → +50.
- `_make_excerpt` (`ai/rag.py:199-235`): picks the first paragraph containing
  any matched term, ±a few lines.
- `retrieve_curriculum_context` (`ai/rag.py:238-328`): scans every `*.txt` in
  `data/curriculum/` (re-read from disk on every query), returns top-K (default
  3) with excerpts.

The corpus is roughly 17 plaintext lesson files under
`data/curriculum/grade6_unit4_lesson*_*.txt`.

### Failure modes of the current approach

- **Synonymy** is brittle: only what is hand-listed in `ALIASES` and
  `concept_boosts` is recognized. "Halve" / "split into pieces" / "share
  equally" will not match `divide`.
- **Paraphrase** breaks scoring entirely (e.g. "how many quarter cups in
  3/4" vs the curriculum's framing).
- **Stop-list is overzealous** — `how`, `what`, `find`, `show`, `explain`,
  `using` are stripped, so question intent is lost.
- **Per-query disk re-read** of every `.txt` file with re-tokenization
  (`ai/rag.py:275-302`) is wasteful as the corpus grows.
- **Hand-tuned boosts are fragile**: each new lesson likely needs another
  `concept_boost` rule.
- **No cross-document concept linking** — there's no notion that
  "lesson 11 algorithm to divide fractions" relates to "lesson 10 dividing by
  unit fractions" beyond shared keywords.

### Expansion options (recommended in order)

1. **BM25 baseline (cheapest immediate win).** Replace the hand-rolled scorer
   with `rank_bm25` over the same `*.txt` files. Pre-tokenize once on startup,
   cache in memory. Removes the per-query disk scan and most of `_score_document`
   without introducing new dependencies on embeddings.

2. **Embedding-based dense retrieval (recommended primary).**
   - **Chunking:** split each lesson `.txt` into ~300-token chunks on
     paragraph/heading boundaries, preserving `{lesson_id, lesson_title,
     section}` metadata. Most lessons are short, so 2–6 chunks per file.
   - **Embedding model:** OpenAI `text-embedding-3-small` (cheap, 1536-dim) is
     the natural fit since `OPENAI_API_KEY` is already configured. If you'd
     rather avoid a second OpenAI dependency, `BAAI/bge-small-en-v1.5` (384-dim)
     via `sentence-transformers` runs locally on CPU.
   - **Index:** for ~17 lessons / ~100 chunks the index is tiny — `numpy`
     cosine over a stacked matrix is enough; no need for FAISS yet. Persist as
     `data/curriculum_index.npz` + a JSON sidecar of chunk metadata. Build at
     startup if the source files are newer than the index.
   - **Query path:** embed `query` once, top-K (k=5) by cosine, then dedupe by
     `lesson_id` so the context block doesn't repeat one lesson.

3. **Hybrid retrieval (recommended final state).** Take top-K from BM25 and
   top-K from dense, merge with Reciprocal Rank Fusion (`score = Σ 1/(60+rank)`),
   keep top-3. Hybrid consistently beats either signal alone on small,
   jargon-heavy corpora like a math curriculum — BM25 catches exact term
   matches ("reciprocal", "prism"), dense catches paraphrase ("flip the second
   fraction" ↔ "reciprocal").

4. **Light reranking (optional, if quality still lags).** Pass the top-10
   hybrid hits through a cross-encoder (e.g. `cross-encoder/ms-marco-MiniLM-L-6-v2`)
   to pick the final 3. Worth it only after (2)+(3) are in place and you
   measure failures on real student questions.

5. **Drop the special-case lesson-number boost into metadata.** If the query
   contains `lesson 11`, filter the candidate pool to chunks where
   `metadata.lesson_id == 11` before scoring. Cleaner than the current +50
   string-match in `_score_document`.

6. **Cache + warm start.** Build the index once at startup
   (`config.py` or `app.py`) and inject the retriever, instead of re-reading
   files in `retrieve_curriculum_context` on every keystroke.

### Minimal migration plan

- Keep `retrieve_curriculum_context(query) -> {"context": str, "matches":
  [{source, score, matched_terms, excerpt}]}` as the public surface — the UI
  in `ui/student.py:583-614` and the call site in
  `ai/tutor_chain.py:172-173` depend on this shape.
- Internally swap the scorer: build a `CurriculumIndex` class that lazily
  embeds at first use and exposes `.search(query, k=3)`. Map dense hits back
  into the same `matches` dict, populating `matched_terms` with the lexical
  intersection so the curriculum-grounding sidebar (`ui/student.py:601-612`)
  keeps working.
- Add `numpy` + an embedding client to `requirements.txt`. If staying inside
  the OpenAI stack, `from openai import OpenAI; client.embeddings.create(...)`
  avoids pulling in `sentence-transformers`.

This change is well-scoped: one file (`ai/rag.py`), no UI changes required,
no schema changes downstream.

## 7) Would a multi-agent architecture make this more intelligent?

**Recommendation:** yes, but as **specialized agents behind a thin
coordinator**, not a heavyweight framework. The four concerns (grounding,
signals, visuals, summaries) already have distinct contracts and can fail /
improve independently — that is exactly the shape that benefits from
per-task agents. Main tradeoff: extra latency + token cost + orchestration
complexity, so introduce agents only where the current hardcoding is the
limiting factor.

### Per-domain agent shape

| Concern | Agent | Model class | Key inputs | Why it beats today |
|---|---|---|---|---|
| Curriculum grounding | **Retriever agent** with tools `bm25_search`, `dense_search`, `lookup_lesson(n)` | small LLM (`gpt-4.1-mini` or `claude-haiku-4-5`) + embedding model | student turn, recent history | Picks the right retrieval strategy per query (lesson# → filter; paraphrase → dense; jargon → BM25); replaces the regex/alias hacks in `ai/rag.py` |
| Signal extraction | **Diagnostician agent** with structured output | `gpt-4.1-mini` at T=0 | last 6 turns + retrieved curriculum + prior `signals_history` | Today's signal call sees only chat (`ai/signals.py:25-29`); giving it curriculum + history catches drifting misconceptions and grounds `next_support` in the actual lesson |
| Mastery update | **Keep as code, not an agent** | — | signal dict | Heuristic in `memory/student_memory.py:250-312` is fine; an LLM here adds noise without value |
| Visual planning | **Visual planner agent** → emits `(visual_type, visual_data)` | `gpt-4.1-mini` | student turn + diagnostician's concept | Replaces the regex fallback in `ui/student.py:283-405`; lets you add new visual types by extending the prompt + a renderer, not the regex |
| Visual rendering | **Keep PIL renderers** OR add VLM "critic" | — / VLM (`claude-opus-4-7` vision) | rendered PNG + caption | Deterministic PIL is correct for math — a VLM critic can verify the image actually shows what the caption claims before sending to the student |
| Teacher insights | **Teacher insight agent** | `gpt-4.1-mini` | full `signals_history`, `mastery_history`, recent `sessions` | Replaces the seeded `insights` list in `memory/student_memory.py:398-420` with grounded, pattern-based observations across sessions |
| Parent summary | **Parent summary agent** (separate prompt, plain-English tone) | `gpt-4.1-mini` | weekly window of signals/mastery | Replaces the templated string-stitching in `update_weekly_summary` (`memory/student_memory.py:61-120`) |

### Coordinator

A thin coordinator (a function, not a framework) runs:

```
retrieve → diagnose → tutor reply → plan visual → render → [async] update signals/mastery
```

Teacher / parent agents run on-demand (dashboard open, session end, weekly
digest) — not per turn — so they don't add chat latency.

### What to build first

1. **Diagnostician agent that sees curriculum + history.** Biggest quality
   lift, ~1 file change in `ai/signals.py`.
2. **Replace `ai/rag.py` with hybrid BM25 + dense retrieval.** Pure
   refactor; no agent needed yet — it becomes a tool the diagnostician /
   tutor / planner can call.
3. **Visual planner agent** replacing the regex fallback in
   `ui/student.py:283-405`.
4. **Teacher + parent summary agents** running on session end / weekly.

### Cautions

- Every agent adds a round-trip. Keep **per-turn** agents to 2 (retriever +
  tutor); push diagnostician, visuals, and analysis to parallel / async
  where possible.
- **Don't** agent-ify the mastery delta math in
  `memory/student_memory.py:250-312` or the PIL rendering in
  `ai/visual_renderer.py` — determinism is a feature there.
- Start with prompts + structured output (`gpt-4.1-mini` + JSON schema). Do
  not adopt LangGraph / CrewAI until the simple version is proven and you
  actually need conditional routing or checkpointing.

See `agent_architecture.md` for the full multi-agent design that
implements this recommendation.

## 8) Engagement determination — needs improvement

How engagement is set today, and why it is one of the weakest signals in
the pipeline.

### How it works now

1. **Set by the LLM, not computed.** The signal-extraction prompt
   (`ai/signals.py:7-18`) asks `gpt-4.1-mini` (T=0) to return one of four
   labels:
   `"engagement": "one of: Active, Moderate, Low, Needs scaffold"`.
2. **Only input is recent chat.** Just the last 6 messages
   (`ai/signals.py:25-29`). No curriculum context, no prior signals, no
   hint counter, no timing or interaction-frequency data.
3. **No rubric.** The prompt never defines what distinguishes Active vs
   Moderate vs Low vs Needs scaffold — the model relies on training priors.
4. **Consumed by substring match downstream** (`memory/student_memory.py:286-307`):

   | Substring in label | Mastery delta |
   |---|---|
   | "active" / "engaged" / "persistent" | +1 |
   | "needs scaffold" / "frustrated" / "disengaged" / "low" | −1 |
   | anything else (incl. "moderate") | 0 |

### Problems

- **"Moderate" is a dead label.** It is in the allowed values but the
  consumer never matches it. The model is rewarded for picking the safe
  middle ground, silently neutralizing the signal.
- **No rubric → inconsistent labels.** Different sessions can tag the same
  behavior differently because the model has no shared definition.
- **Behavioral signals are ignored.** Quick-action button clicks
  (`ui/student.py:111-138`), message length, response latency, number of
  clarifying questions, retries, and idle time all exist in UI state but
  never reach the classifier.
- **Single-snapshot, no trend.** Engagement is recomputed per turn from
  the latest 6 messages with no memory of the previous label, so it can
  flap (Active → Low → Active) without any real change in the student.
- **Only chat is available to the LLM.** It cannot see hint usage even
  though `hints_used` is being extracted in the same call — so a student
  who burned three hints can still be labelled "Active" if their last
  message sounds upbeat.

### Improvements (cheapest first)

1. **Add a rubric to the prompt.** One sentence + an example per label.
   Removes most label drift at zero infrastructure cost.
   ```
   - Active: attempts steps before asking, asks follow-up questions,
     responds with reasoning. Example: "I think I need to flip 1/8 — is that right?"
   - Moderate: answers when asked but doesn't volunteer reasoning.
   - Low: very short replies, no questions, no attempts.
   - Needs scaffold: says "I don't get it", asks for the answer, gives up.
   ```
2. **Feed behavioral features to the classifier.** Pass alongside the
   chat: number of quick-action clicks, average user-message length, hint
   button count, idle time since last user message, previous-turn
   engagement label. These are all already in `st.session_state`.
3. **Make "Moderate" do something** — either remove it (collapse to
   three labels) or give it a small positive/negative weight in the
   mastery delta. A label with no downstream effect is dead code.
4. **Smooth across turns.** Either (a) require two consecutive turns of
   the same label before flipping in the persisted signal, or (b) take
   the majority over the last N turns. Reduces flapping in
   `signals_history`.
5. **Promote to a small dedicated classifier.** Once the rubric and
   features above are in place, this becomes a tiny, deterministic
   structured-output call — split it out of the main signal extraction
   so the rubric can evolve independently of the misconception detector.
6. **Treat it as a soft probability, not a hard label.** Have the model
   emit `{active: 0.6, moderate: 0.3, low: 0.1, needs_scaffold: 0.0}`
   instead of a single string. The mastery delta then becomes a weighted
   sum, removing the all-or-nothing keyword-match behavior.

This fix is well-scoped: prompt change in `ai/signals.py:7-18`, a small
extension of the input payload built in `ai/signals.py:25-29`, and a
keyword-table update in `memory/student_memory.py:286-307`. No UI or
schema changes required.

## 9) Skill-bucket matching — hardcoded, should not be

The concept → skill-bucket mapping used by `update_mastery` is keyword-only
today and is misaligned with the actual curriculum on disk.

### How it works now

`_match_mastery_skill` in `memory/student_memory.py:185-247`:

- A 4-row hardcoded keyword table maps `concept` strings to one of four
  fixed skill buckets:

  | Skill bucket | Keywords |
  |---|---|
  | Fraction comparison | "compare fraction", "comparing fraction", "which fraction", "larger fraction", "smaller fraction" |
  | Equivalent fractions | "equivalent fraction", "equivalent fractions" |
  | Fraction addition | "fraction addition", "adding fraction", "add fraction", "common denominator", "least common denominator", "least common multiple" |
  | Word problems | "word problem", "story problem", "real-world problem" |

- Substring-match fallback at lines 238-245.
- The four buckets themselves are seeded in `_default()`
  (`memory/student_memory.py:374-379`) — **so the system literally cannot
  track anything outside those four.**

### Problems

- **Schema-vs-curriculum mismatch.** The curriculum on disk
  (`data/curriculum/grade6_unit4_*.txt`) is Unit 4 = **fraction division**,
  but there is no fraction-division bucket. Today's curriculum and today's
  mastery schema are misaligned by construction.
- **Silent no-op on miss.** LLM concepts like "Dividing fractions by unit
  fractions" or "Reducing fractions to lowest terms" match nothing →
  `update_mastery` returns `{updated: False}` and the session leaves no
  trace on mastery. The student practiced, but their score didn't move.
- **Paraphrase-brittle.** Anything the keyword table doesn't anticipate is
  invisible, exactly like `ai/rag.py`.
- **Buckets are baked into Python.** A teacher cannot add or edit skill
  buckets without a code change.

### Solution options (cheapest first)

1. **Embedding similarity to bucket descriptions** *(recommended)*. For
   each bucket, author a one-sentence description ("Equivalent fractions:
   recognizing that 2/4 = 1/2 = 4/8 by scaling numerator and denominator
   together"). At classify time: embed the LLM `concept`, cosine against
   the bucket descriptions, argmax above a threshold. Deterministic,
   ~5 ms per call, no new buckets get invented at runtime. Reuses the
   `text-embedding-3-small` index already proposed for retrieval.

2. **LLM classifier with a fixed enum.** Single structured-output call:
   "given concept X, return one of [Fraction comparison, ..., None]".
   Handles paraphrase better than embeddings on very short strings, but
   adds a round-trip and introduces nondeterminism in a number that
   parents and teachers see.

3. **Hybrid: keyword first, LLM fallback only on miss.** Keeps current
   speed and determinism for common cases; spends a token call only when
   rules don't match. Worth it if you want to keep keyword auditability.

4. **Full "Skill Taxonomist" agent that can also propose new buckets.**
   When concept matches nothing, the agent suggests a new bucket with
   definition + initial score. Powerful but introduces **schema drift**
   — the set of skills the teacher sees changes over time without their
   input. Only viable behind an explicit approval step in the teacher UI.

### Recommendation

**Option (1): embedding-based bucket matching.** Only one that gives
all three of (a) paraphrase tolerance, (b) determinism — same concept
always maps to the same bucket, (c) no agent on the hot path. It also
composes cleanly with the curriculum-grounding embedding work already
on the roadmap (same model, same index infrastructure).

**Separately, fix the bucket list itself.** Whatever classifier ships:

- Add `Fraction division` (and likely `Fraction multiplication`,
  `Volume of prisms`, `Area with fractional side lengths`) to match the
  actual curriculum in `data/curriculum/`.
- Move buckets out of `_default()` in `memory/student_memory.py:374-379`
